# Why IMake

## Origin

Currently, IMakeCore is defined as: `IMakeCore is a package management system for C++ source code based on cmake and qmake`.

The initial reason for writing IMakeCore was the increasing amount of code and functionality in IWebCore, which allowed for splitting and recombination of different directional features. At the beginning, I encapsulated the pri packages from qmake on my own. If a certain feature was needed, I would manually write the related pri into the configuration to import it.

Additionally, compared to traditional package management, they typically compile code into dynamic or static libraries before introducing them into a project. However, this approach is not applicable to the classes in IWebCore, which heavily rely on static initialization for dependency injection functionality. Compiling into static or dynamic libraries would lead to ODR (One Definition Rule) related issues.

Another point is that IWebCore is not merely a collection of several features. In the planning of IWebCore, it includes not only the current packages but also additional packages for functional extensions. For example, in IHttpCore, there will be extensions such as CORS, cache, ranges, session, static file services, and a series of other HTTP enhancement services. Outside of IHttpCore, there is also a need to encapsulate databases like Redis and MongoDB, as well as RPC and WS. These will generate a large number of packages. If these packages can be managed, they would greatly facilitate users' development.

In contrast, we can take Maven/Gradle as an example for Spring Framework. Thanks to package management systems like Maven, users can easily integrate these libraries into their projects with almost no cost, expanding their functionality and scope of use. Similarly, tools like pip/pypi are used for platforms like Django.

The initial purpose of writing IMake was to manage the libraries used in the IWebCore framework, transforming IWebCore from a collection of libraries into a platform through these extensions.

## Development

When IMake was initially written, it was defined using pri files to specify libraries and import them. During iterative development, its functionality gradually expanded.

First, support was added for cmake beyond qmake. Users no longer needed to manually write pri or cmake files to define a library. Instead, I defined a series of cmake and qmake functional functions to automatically scan the entire package content and configure it. The `autoScan` field in package.json was introduced for this purpose. It should be noted that users can still write pri and cmake files to import libraries, allowing them to define additional features as needed.

Second, the IPubCore website was established. This website allows users to upload, search, manage, and distribute user-defined packages. Its functionality is similar to sites like PyPI, Maven, or npm. Users can upload their packages to share them with others. This website was built from scratch using the IHttp and IRdb sub-frameworks. The frontend uses React, but most of the code was generated by AI, with only necessary modifications made by hand.

Third, users often need to perform operations on their own machines. To address this, I developed the IPubCmd tool, initiated by the `ipc` command. This tool is built on the ICmdCore sub-framework. Users can easily integrate IMakeCore packages into their existing projects and search for and download various packages using ipc.

Recently, I restructured the entire system to enrich the packages and encourage user uploads. This initiative aims to lower the barrier for users to upload packages. Previously, a package name could only be uploaded once. Now, I added a publisher constraint before the package name, allowing different users to encapsulate, upload, and manage the same package. For example, a zip package can be encapsulated by User A, User B, or even modified by User C (changing the publisher field) and re-uploaded.

Regardless of who uploads the package, the author and license information must remain intact to comply with open-source protocols. This means that while the uploader and author can be different, the original author cannot be changed, and anyone can upload as long as they adhere to open-source protocols.

As a result, IPubCore will host a large number of packages, which may vary in quality. After addressing the issue of insufficient packages, I will implement sorting algorithms to prioritize excellent packages. Sorting can be based on factors like user stars and download counts. Exceptional packages or those uploaded by the original author will be designated as `official packages`, allowing users to omit the publisher field when importing them. Official packages will appear higher in default searches.
