# Package遗留问题

这里汇总一些 Package 悬而未决的问题，把问题思考清楚。

## 命名问题
1. 之后的package group 名称不变， name 名称后面添加 @version 来解决版本识别的问题。
2.  具体的package .pri package .cmake 的名称不变。还是不需要带上 version 版本号。
3.  这个不会造成冲突的。
4. 针对于 asio 和 json 库而言，从新封装一下，把版本号一同封装起来，免得像现在这个样子。

## Package 版本问题

1. 安装版本的* 表示最新版本，但是我们现在没有最新的版本，连服务器都没有，所以还是老老实实的使用具体的版本。
2. 之后的版本设置按照 node.js的设置来做。

## package lib 自动生成 .pri 文件，.cmake 文件的问题。

> 就是在我们目前的基础上，需要有一些脚本可以自动生成 pri， cmake 文件，之后也包括其他类型的文件。

1. 位置
    1.  这里这些内容的配置位置可以在 lib.json 文件中配置
    2. 在之后的文件中能够解析这些内容。
2. 种类
     1. sources，
           1. 这个可以是文件列表，也可以是一个文件夹的名称。具体做一个区分。
           2.  如果内容是字符中，则是文件，如果是对象，则根据对象的类型做一个区分。
     2. headers，
               1. 同上
   3. resources`
           1. 同上 
   4. includes
             1. 这里是相对于当前顶层目录的位置，之后在具体转换的时候在行决定。
     5. libs
             1. 具体的库，这个库可以之后在行考虑。
     6. definition。
               (1)  key， value pair
               (2) 如果是value 是字符串，还需要一次转义才可以。
               (3) 如果为 null， 则直接定义，没有具体的值
               (4) 其他的类型有 bool， number， 和 字符串和 真字符串类型。

3. 额外的内容
    1. 这个用户可以选择是否生成某一类的文件，也就是说，可以自定义这个的文件，而不需要程序自动生成。
    2. 用户可以选择，在生成的文件后面添加一些额外的内容。
        1. 内容可以是字符串定义在 json 文件里面，也可以是一个额外的文件。
